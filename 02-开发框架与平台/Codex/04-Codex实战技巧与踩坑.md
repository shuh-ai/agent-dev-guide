# Codex 桌面端实战技巧与踩坑

> 本文档专注于 Codex App 桌面端（含 Codex++ 增强）的使用技巧，不涉及 CLI 命令行操作。

---

## 1. 常见桌面端工作流

### 1.1 新功能开发

在 Codex 桌面端中，你可以直接用自然语言描述需求，Codex 会自动生成代码：

**典型流程：**
1. 打开 Codex App，选择对应的本地项目
2. 在对话框中描述功能需求
3. Codex 生成代码并展示变更预览
4. 审阅后确认应用或要求修改

**示例 Prompt：**
```
在 src/components/ 下创建一个用户管理表格组件：
- 使用 React + TypeScript
- 支持排序、搜索、分页
- 从 /api/users 接口获取数据
- 使用 Ant Design 组件库
```

**技巧：** 提供明确的技术栈和目录约束，减少 Codex 的猜测。

### 1.2 代码重构

将需要重构的代码直接粘贴到对话中，或引用项目中的文件：

**典型流程：**
1. 在聊天框中粘贴目标代码，或输入文件路径引用
2. 描述重构目标（如：从回调改为 async/await）
3. Codex 生成重构后的代码
4. 对比查看变更，逐个接受或拒绝

**示例 Prompt：**
```
将以下代码重构为 async/await 风格：
- 添加完整的 TypeScript 类型定义
- 增加错误处理和重试逻辑
- 保持原有功能不变

[粘贴代码或引用文件]
```

### 1.3 Bug 调试

Codex 桌面端支持粘贴错误日志和拖拽截图，非常适合调试场景：

**典型流程：**
1. 将报错信息复制粘贴到聊天框
2. 拖拽错误截图到对话窗口（支持图片识别）
3. Codex 分析错误原因并给出修复方案
4. 确认修复后 Codex 直接修改代码

**示例 Prompt：**
```
运行 npm test 报以下错误：

TypeError: Cannot read properties of undefined (reading 'map')
    at UserList (src/components/UserList.tsx:23:18)

请分析错误原因并修复。
```

**技巧：** 截图时裁剪关键区域，保证文字清晰可读，Codex 的 OCR 识别效果更好。

### 1.4 测试生成

让 Codex 为现有代码自动生成测试用例：

**示例 Prompt：**
```
为 src/services/auth.ts 编写完整的单元测试：
- 使用 Vitest 测试框架
- 覆盖正常流程和边界情况
- Mock 外部依赖（数据库、API 调用）
- 覆盖率目标 80% 以上
```

### 1.5 代码审查

通过粘贴 PR diff 或连接代码仓库进行代码审查：

**方式一：粘贴 diff**
```
请审查以下 PR diff，重点关注：
- 潜在的安全问题
- 性能隐患
- 代码风格一致性

[粘贴 git diff 输出]
```

**方式二：引用项目文件**
```
审查 src/api/payments.ts 的代码质量，重点关注：
- 错误处理是否完善
- 是否有 SQL 注入风险
- 事务处理是否正确
```

## 2. Prompt 编写技巧

> 以下技巧适用于 CLI 和桌面端，但桌面端可以利用多轮对话逐步完善。

### 2.1 结构化 Prompt

```
❌ 差的 prompt：
"帮我改一下代码"

✅ 好的 prompt：
"将 src/utils/parser.ts 中的 parseCSV 函数重构：
1. 添加类型定义
2. 处理空行和格式错误
3. 支持自定义分隔符
4. 添加单元测试"
```

### 2.2 提供上下文

```
✅ 好的 prompt：
"这个项目使用 Next.js 14 App Router。在 app/dashboard/page.tsx 中添加
一个数据表格组件，数据从 /api/stats 接口获取，需要支持排序和分页。"
```

### 2.3 约束条件

```
✅ 好的 prompt：
"重构 auth 模块：
- 使用 bcrypt 替代明文密码
- 添加 rate limiting
- 保持现有 API 接口不变
- 不要修改数据库 schema"
```

### 2.4 Prompt 模板

| 场景 | Prompt 模板 |
|------|-------------|
| 新功能 | "在 [位置] 实现 [功能]，使用 [技术栈]，需要 [具体要求]" |
| 重构 | "将 [目标] 从 [旧方式] 改为 [新方式]，保持 [不变的部分]" |
| 调试 | "以下是错误信息 [粘贴]，分析原因并修复" |
| 测试 | "为 [目标] 编写 [测试框架] 测试，覆盖 [场景]" |
| 审查 | "审查 [范围]，关注 [重点]，输出 [格式]" |

## 3. Codex++ 高级功能

### 3.1 插件系统

Codex++ 通过插件系统扩展 Codex App 的能力：

**访问方式：** 点击 Codex++ 菜单 → 插件管理

**插件类型：**

| 插件类别 | 功能说明 | 示例 |
|----------|----------|------|
| 代码分析 | 静态分析、复杂度检测 | ESLint 集成、SonarQube |
| 文档生成 | 自动生成 API 文档 | Swagger、TypeDoc |
| 数据库工具 | 数据库操作辅助 | Prisma Studio 集成 |
| 部署工具 | 容器化、部署辅助 | Docker 配置生成 |

**安装流程：**
1. 打开 Codex++ 菜单 → 插件管理
2. 浏览可用插件或搜索
3. 点击安装，等待依赖下载完成
4. 在插件设置中配置参数

### 3.2 中转注入

中转注入是 Codex++ 的核心功能，允许你将模型请求路由到第三方 API 平台，同时保持会话连续性：

**官方赞助商/中转平台：**

| 平台 | 特点 | 注册地址 |
|------|------|----------|
| **JOJO Code** | Codex++ 官方中转站，稳定接入 | jojocode.com |
| **AIGoCode** | Claude Code + Codex + Gemini 一体化 | aigocode.com |
| **RunAPI** | 150+ 模型，高效稳定 | runapi.co |
| **APIKEY.FUN** | 全球主流大模型，价格低至 7% | apikey.fun |
| **PackyCode** | 稳定高效，支持多种中转服务 | packyapi.com |
| **AIHub2API** | 专注 Codex 中转，高缓存命中 | aihub2api.cloud |

**配置步骤：**
1. 在第三方平台注册并获取 API Key 和 Base URL
2. 打开 Codex++ 管理工具 → 中转注入页面
3. 添加中转配置，填写 Base URL 和 Key
4. 选择配置并应用中转注入
5. 启动 Codex++

**配置示例（~/.codex/config.toml）：**

```toml
model_provider = "CodexPlusPlus"

[model_providers.CodexPlusPlus]
name = "CodexPlusPlus"
wire_api = "responses"
requires_openai_auth = true
base_url = "https://your-api-endpoint.com/v1"
experimental_bearer_token = "sk-..."
```

**注意事项：**
- 中转注入后，官方 ChatGPT 登录态继续负责账号能力和插件入口
- 中转配置只接管模型请求，不影响其他功能
- 清除 API 模式后可回到官方登录态
- 建议在切换前导出当前会话作为备份

### 3.3 会话管理

Codex++ 提供了强大的会话管理功能：

| 功能 | 操作路径 | 说明 |
|------|----------|------|
| 删除会话 | Codex++ 菜单 → 会话管理 → 选择 → 删除 | 支持批量删除 |
| 撤销删除 | 删除后底部出现撤销提示 → 点击"撤销" | 5 秒内可撤销 |
| 导出会话 | Codex++ 菜单 → 导出为 Markdown | 保存完整对话记录 |
| 会话搜索 | 会话列表顶部搜索框 | 按关键词搜索历史会话 |
| 时间线 | 点击会话 → 右上角"时间线"按钮 | 查看对话时间轴 |

### 3.4 会话迁移

Codex++ 支持在普通聊天和本地项目之间迁移会话：

**场景：** 你在普通聊天中讨论了一个方案，现在想在具体项目中执行。

**操作步骤：**
1. 在当前会话中，点击 Codex++ 菜单
2. 选择"将会话移至项目"
3. 选择目标项目目录
4. 会话上下文会自动关联到项目文件

**反向迁移：**
- 在项目会话中，选择"在普通聊天中继续"可以将讨论抽离项目上下文

### 3.5 社区技巧

来自 Codex++ 社区的实用技巧：

1. **分步执行：** 复杂任务拆分为多步对话，每步确认后再继续，效果优于一次性描述所有需求
2. **模板复用：** 将常用的 Prompt 模板保存在文本文件中，需要时直接粘贴
3. **截图标注：** 使用截图工具的标注功能，在截图上圈出问题区域，再拖入 Codex
4. **项目笔记：** 在 AGENTS.md 中记录项目的技术决策，帮助 Codex 理解上下文
5. **增量审查：** 代码审查时只粘贴变更部分（diff），避免 Codex 分析无关代码

## 4. AGENTS.md 最佳实践（桌面端）

AGENTS.md 是 Codex 的项目配置文件，桌面端用户同样需要配置好它。

### 4.1 基本结构

在项目根目录创建 `AGENTS.md` 文件：

```markdown
# 项目概述
这是一个 [项目类型] 项目，使用 [技术栈]。

# 代码规范
- 使用 TypeScript strict 模式
- 函数和变量使用 camelCase
- 组件使用 PascalCase
- 每个函数不超过 30 行

# 目录结构
- src/components/ - React 组件
- src/services/ - 业务逻辑
- src/utils/ - 工具函数
- tests/ - 测试文件

# 禁止事项
- 不要修改 .env 文件
- 不要删除现有的测试文件
- 不要更改数据库 migration 文件
```

### 4.2 多层级配置

| 层级 | 文件位置 | 作用范围 |
|------|----------|----------|
| 全局 | `~/.codex/AGENTS.md` | 所有项目通用设置 |
| 项目 | `项目根目录/AGENTS.md` | 当前项目设置 |
| 子目录 | `src/api/AGENTS.md` | 特定目录的规则 |

### 4.3 桌面端特有技巧

- 在 AGENTS.md 中记录项目使用的 Codex++ 插件配置
- 标注项目依赖的外部服务和 API 地址
- 记录项目的构建和运行方式（供 Codex 参考，而非执行）

## 5. 常见踩坑与解决方案

### 5.1 Codex++ 菜单不显示

| 问题 | 原因 | 解决方案 |
|------|------|----------|
| Codex++ 菜单完全不出现 | 未正确安装 Codex++ | 从 Codex++ 官方 GitHub Releases 下载安装 |
| 菜单灰色不可用 | Codex App 未运行 | 先启动 Codex App，再启动 Codex++ |
| 安装后需要重启 | 进程未完全加载 | 关闭所有 Codex 相关进程后重新启动 |

**关键：** 必须从 Codex++ 快捷方式启动，而不是直接启动原版 Codex App。

### 5.2 插件后端断开连接

| 问题 | 原因 | 解决方案 |
|------|------|----------|
| 插件显示"已断开" | 后端服务未启动 | 打开 Codex++ Manager → 诊断 → 重启后端 |
| 插件加载失败 | 依赖缺失 | 在诊断页面检查依赖状态并修复 |
| 插件无响应 | 端口冲突 | 检查端口占用情况，修改插件端口配置 |

**诊断步骤：**
1. 打开 Codex++ Manager
2. 进入"诊断"页面
3. 查看"插件后端状态"
4. 如显示异常，点击"重启后端服务"

### 5.3 认证问题

| 问题 | 原因 | 解决方案 |
|------|------|----------|
| 提示 API Key 无效 | Key 过期或格式错误 | 在 Codex++ Manager 中重新配置 API Key |
| 登录后仍提示认证失败 | Token 缓存损坏 | 删除 `~/.codex/auth.json` 后重新登录 |
| 多账户切换失败 | 缓存冲突 | 退出 Codex，清除缓存后重新登录 |
| Rate limit 错误 | API 配额用尽 | 升级计划或切换到其他 Provider |

### 5.4 模型切换问题

| 问题 | 原因 | 解决方案 |
|------|------|----------|
| 切换模型后效果无变化 | 缓存未刷新 | 关闭当前会话，新建会话测试 |
| 特定模型不可用 | Provider 不支持 | 检查当前 Provider 的模型列表 |
| 切换后报错 | API 格式不兼容 | 确认目标模型与当前 Provider 兼容 |

### 5.5 会话数据防丢失

**预防措施：**
1. **定期导出：** 重要对话及时导出为 Markdown
2. **Git 提交：** Codex 修改的代码及时提交到 Git
3. **备份目录：** 定期备份 `~/.codex/sessions/` 目录
4. **避免强制关闭：** 正常退出 Codex，避免强制结束进程

**数据恢复：**
- 如果误删会话，5 秒内点击"撤销"可恢复
- 超过 5 秒的删除无法通过 UI 恢复，需从备份目录手动恢复

## 6. FAQ

### Q1: Codex App vs Codex Web vs CLI 有什么区别？

| 维度 | Codex App（桌面端） | Codex Web | CLI |
|------|---------------------|-----------|-----|
| 界面 | 原生桌面应用 | 浏览器 | 终端命令行 |
| 安装 | Codex++ 安装包 | 无需安装 | npm 安装 |
| 文件操作 | 直接访问本地文件 | 需要手动上传 | 命令行操作 |
| 图片支持 | 拖拽截图 | 上传图片 | `--image` 参数 |
| 增强功能 | Codex++ 全功能 | 基础功能 | 基础功能 |
| 插件系统 | ✅ 支持 | ❌ 不支持 | ❌ 不支持 |
| 离线使用 | 部分支持 | ❌ 不支持 | ❌ 不支持 |
| 推荐场景 | 日常开发 | 快速体验 | 自动化脚本 |

### Q2: 可以同时使用多个 OpenAI 账户吗？

可以。通过 Codex++ Manager 配置多个 Provider：
1. 打开 Codex++ Manager → Provider 设置
2. 添加多个账户的 API Key
3. 切换时选择对应账户即可
4. 每个账户的使用额度独立计算

### Q3: 如何备份会话？

**手动导出：**
- 在会话中点击 Codex++ 菜单 → "导出为 Markdown"
- 选择保存位置

**自动备份：**
- 会话数据存储在 `~/.codex/sessions/` 目录
- 可以使用文件同步工具（如坚果云、OneDrive）同步该目录
- 建议定期检查该目录大小，清理不需要的旧会话

### Q4: Codex++ 是否安全？

Codex++ 是开源项目，安全性说明：
- **开源透明：** 源代码公开可审计
- **通信方式：** 通过 CDP（Chrome DevTools Protocol）与 Codex App 通信，仅限本地
- **数据安全：** 不会上传你的代码到 Codex++ 服务器
- **下载安全：** 务必从官方 GitHub Releases 下载，避免使用第三方渠道

### Q5: 如何更新 Codex++？

1. 打开 Codex++ Manager
2. 检查是否有新版本提示
3. 点击"更新"按钮，等待下载完成
4. 重启 Codex App 和 Codex++

**手动更新：**
- 前往 Codex++ GitHub Releases 页面
- 下载最新版本安装包
- 覆盖安装即可，会话数据不会丢失

### Q6: 桌面端遇到问题如何获取帮助？

1. **Codex++ 诊断工具：** Manager → 诊断 → 查看状态
2. **GitHub Issues：** 在 Codex++ 仓库提交 Issue
3. **社区讨论：** 加入 Codex++ Discord / 论坛社区
4. **日志查看：** `~/.codex/logs/` 目录下的日志文件
