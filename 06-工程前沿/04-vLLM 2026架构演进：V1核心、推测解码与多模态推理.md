## 二、V1 核心架构

### 2.1 为什么重写

V0 的问题：
- **不对称架构**：Scheduler 和 Worker 0 捆在一起，增加复杂度
- **CPU 瓶颈**：调度逻辑在 CPU 上执行，成为延迟瓶颈
- **分布式逻辑耦合**：Worker 代码中混杂了分布式通信逻辑
- **新模型适配困难**：每加一个新模型都需要大量适配工作

### 2.2 V1 的核心设计原则

**原则一：GPU-first**

所有热路径都放在 GPU 上执行，CPU 只做冷路径（配置、初始化）：

```
V0 路径：
  CPU: 调度决策 → CPU→GPU 数据传输 → GPU: 计算 → GPU→CPU 结果回传

V1 路径：
  GPU: 调度缓存 + 计算 + 结果（全程 GPU，零同步）
```

**原则二：对称架构**

Scheduler 和 Worker 分离到不同进程，通过增量更新（diff）通信：

```
V0: Scheduler + Worker 0 在同一进程 → Worker 1..N 在独立进程
V1: Scheduler 在独立进程 → Worker 0..N 都在独立进程（对称）
```

Worker 端缓存请求状态，每步只传输增量更新（diff），最小化进程间通信。

**原则三：统一分布式抽象**

无论单卡还是多卡，Worker 的代码路径完全相同：

```python
# 单卡推理
VLLM_USE_V1=1 vllm serve meta-llama/Llama-3.3-70B-Instruct

# 4卡张量并行
VLLM_USE_V1=1 vllm serve meta-llama/Llama-3.3-70B-Instruct -tp 4

# Worker 代码完全一样，分布式逻辑被抽象掉了
```

### 2.3 性能提升

根据 vLLM 官方数据和社区评测：
- V1 vs V0：开箱即用 **2x 吞吐量提升**
- 在已优化的 decode 路径上，每个百分点的提升都很关键
- 零同步、零 CPU 开销的设计为未来硬件预留了空间

## 三、推测解码（Speculative Decoding）

### 3.1 基本原理

LLM 自回归生成的瓶颈：每生成一个 token 都要等上一个完成。

```
传统自回归：
  token_1 → 等待 → token_2 → 等待 → token_3 → 等待 → ...

推测解码：
  草稿模型快速猜测 token_1,2,3,4,5 → 目标模型一次性验证 → 接受/拒绝
```

**核心权衡**：用额外计算换更少的内存移动。在低 batch size 时（内存瓶颈），效果最好。

### 3.2 EAGLE 3.1（2026.5.26）

**三方协作**：EAGLE 团队 + vLLM 团队 + TorchSpec 团队

**核心创新**：
- **FC 归一化**：提升推测解码的鲁棒性
- **Post-norm 隐藏状态反馈**：改善草稿质量
- **TorchSpec 训练支持**：标准化的训练流程
- **配置驱动兼容**：兼容 EAGLE 3 的检查点

**使用方式**：

```bash
VLLM_USE_V1=1 vllm serve meta-llama/Llama-3.3-70B-Instruct \
  --speculative-config '{
    "model": "yuhuili/EAGLE3-LLaMA3.3-Instruct-70B",
    "num_speculative_tokens": 3,
    "method": "eagle3",
    "draft_tensor_parallel_size": 1
  }'
```

**性能数据**：在低请求率场景下，延迟降低最高 **2.5x**。

### 3.3 P-EAGLE（2026.3.13）

**并行推测解码**：在一次前向传播中生成多个草稿 token。

- 预训练的草稿头（drafter heads）
- 配置支持
- 在 B200 GPU 上比 EAGLE-3 更快

### 3.4 Speculators v0.5.0（2026.5.28）

推测解码的统一框架：
- **DFlash 支持**：新的推测解码方法
- **在线训练**：运行时自适应训练草稿模型
- 与 vLLM 和 LLM Compressor 深度集成

### 3.5 推测解码的实际效果

根据 Snowflake Arctic Inference 的评测数据：

| 工作负载 | 无推测 | N-gram | EAGLE | Arctic LSTM+Suffix |
|---------|--------|--------|-------|--------------------|
| ShareGPT | 76 tok/s | 91 tok/s | 102 tok/s | 179 tok/s |
| HumanEval | 77 tok/s | 100 tok/s | 112 tok/s | 217 tok/s |
| SWE-Bench | 76 tok/s | 175 tok/s | 123 tok/s | 302 tok/s |

**注意**：推测解码的效果高度依赖请求内容和请求率。低并发时提升明显，高并发时（计算瓶颈）提升有限甚至可能下降。

## 四、多模态推理

### 4.1 架构演进

2026年的前沿 LLM 通常有四个组件：

```
多模态编码器 → 语言主干 → 稀疏/线性注意力 → 输出头
(视觉/音频)    (Transformer)   (高效注意力层)
```

**vLLM 的处理**：
- **Encoder 解耦**：将多模态编码器从语言主干中分离，独立优化
- **Encoder 异构部署**：编码器和语言主干可以放在不同的硬件上
- **流水线并行**：编码器和语言主干可以流水线执行

### 4.2 vLLM-Omni

- 统一处理文本、图像、视频输入
- Diffusion Cache 加速
- 多模态编码器的独立优化

### 4.3 实际性能

在 GTC 2026（2026年3月）的演讲中，vLLM 团队展示了：
- DeepSeek V3.2 在 GB300 上的性能突破
- Wide Expert Parallelism (Wide-EP) 在大规模 serving 中的成熟度
- 多模态 encoder 解耦带来的吞吐量提升

## 五、其他重要更新

### 5.1 KV Cache 管理

- **PagedAttention v3**：分层内存管理，支持 TB 级模型分布式推理
- **外部 KV Cache**：vLLM x Novita AI 的 PegaFlow，生产级外部缓存
- **KV Offloading Connector**：更智能的内存传输策略

### 5.2 推理框架支持

- **Prefill-Decode 解耦**：将预填充和解码阶段分离到不同节点
- **异构计算集群**：支持 CPU + GPU + NPU 混合调度
- **Blackwell 优化**：针对 NVIDIA B200/GB300 的深度优化

### 5.3 Native RL APIs（2026.5.28）

vLLM 原生支持强化学习训练的推理：
- 直接在 vLLM 上运行 RL 训练的 rollout
- 无需额外的推理服务

### 5.4 Session-Aware Agentic Routing（2026.6.2）

最新的更新——为 Agent 系统设计的会话感知路由：
- 保持长对话的模型一致性
- 安全的模型切换边界
- 前缀缓存感知的切换成本计算
- 可重放的 Trace

## 六、vLLM vs 其他推理框架

### 6.1 2026年推理框架对比

| 框架 | 核心优势 | 适用场景 |
|------|---------|---------|
| **vLLM** | 生态完整、社区活跃、V1 架构 | 通用推理、多模型、Agent 系统 |
| **SGLang** | 推测解码 2.0、SGLang Compiler、多模态原生 | 高性能推理、结构化输出 |
| **TensorRT-LLM** | NVIDIA 深度优化、边缘设备 | NVIDIA 硬件、边缘部署 |
| **Ollama** | 极简部署、本地运行 | 本地开发、快速原型 |

### 6.2 选型建议

```
你的场景是什么？
│
├── 生产环境、多模型、需要路由 → vLLM + Semantic Router
├── 极致性能、NVIDIA 硬件 → TensorRT-LLM
├── 结构化输出、编程 DSL → SGLang
├── 本地开发、快速原型 → Ollama
└── 边缘设备、资源受限 → TensorRT-LLM / ONNX Runtime GenAI
```

## 七、趋势展望

### 7.1 推理框架成为 AI 基础设施

2026年的推理框架不再是"性能优化工具"，而是完整的 AI 基础设施：
- 智能路由（语义路由）
- 安全过滤（PII、越狱检测）
- 可观测性（Trace、Metrics）
- 成本管理（模型路由、缓存）

### 7.2 推测解码成为标配

随着模型越来越大、推理成本越来越高，推测解码从"锦上添花"变成"必须有"：
- EAGLE 3.1、P-EAGLE、Speculators 形成完整的推测解码生态
- 在线训练让草稿模型可以自适应
- 和 KV Cache 优化、量化等技术组合使用

### 7.3 多模态推理的标准化

2026年的前沿模型几乎都是多模态的（GPT-5.4、Gemini 3.1、Claude Opus 4.7），推理框架必须原生支持：
- Encoder 解耦成为标准架构
- 多模态缓存（如 Diffusion Cache）降低重复计算
- 跨模态注意力的高效实现

### 7.4 RL 训练与推理的融合

vLLM 的 Native RL APIs 标志着一个趋势：训练和推理不再是分离的系统。同一个引擎同时服务在线推理和训练 rollout，大幅降低基础设施复杂度。
