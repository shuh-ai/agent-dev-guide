# 上下文工程全景：从 Prompt Engineering 到 Context Engineering

> **一句话定位**：Prompt Engineering 教你怎么写指令，Context Engineering 教你怎么管理模型看到的**全部信息**。

---

## 一、为什么需要上下文工程

### 1.1 一个真实的失败场景

假设你在做一个智能客服 Agent，用户问：

> "我上周买的那台洗衣机，安装师傅说要换个水龙头，这个费用能报销吗？"

你的 Prompt 写得很好，但 Agent 的回答一塌糊涂。为什么？

因为它没有看到：
- 用户上周的订单记录（需要检索）
- 用户之前和客服的对话历史（需要记忆）
- 售后报销政策文档（需要 RAG）
- 安装服务的费用标准（需要工具调用）

**Prompt 只是冰山一角，模型真正需要的是完整的上下文。**

### 1.2 Karpathy 的定义

Andrej Karpathy（前 OpenAI/Tesla AI 负责人）在 2025 年提出了一个被广泛引用的定义：

> "上下文工程是为下一步填充上下文窗口中**恰好正确信息**的精妙艺术与科学。"

关键词是"恰好正确"——不是把所有信息都塞进去，而是**精准组装**模型此刻最需要的信息。

### 1.3 核心类比

```
LLM          = CPU（计算单元）
上下文窗口    = RAM（内存）
上下文工程    = 操作系统的内存管理
```

你的电脑有 32GB 内存，但不是把所有文件都加载到内存里——操作系统会根据当前任务，动态加载最需要的数据。上下文工程做的就是同样的事情。

## 二、Context Engineering vs Prompt Engineering

很多人问：这不就是换了个名字吗？

不是。它们的关系是这样的：

```
Context Engineering（上下文工程）
├── Prompt Engineering（系统指令设计）    ← 你已经会的
├── RAG（检索增强）                       ← 知识注入
├── 记忆管理（短期/长期）                  ← 历史累积
├── 工具输出整合                          ← 实时信息
├── 上下文压缩                           ← 信息精简
└── 多 Agent 隔离                        ← 信息分区
```

### 对比表

| 维度 | Prompt Engineering | Context Engineering |
|------|-------------------|---------------------|
| **关注点** | 怎么措辞单条指令 | 模型此刻需要看到什么？如何动态组装？ |
| **范围** | 系统提示词 | 系统提示 + 工具定义 + 检索知识 + 记忆 + 对话历史 + 工具输出 |
| **时间维度** | 单次请求 | 跨会话、长任务持续管理 |
| **适用场景** | 一次性任务、内容生成 | 对话 AI、Agent、编码助手、文档分析 |
| **类比** | 写一句话 | 设计整个信息系统 |
| **技能门槛** | 中等 | 高（需要系统设计能力） |

### 关键洞察

Karpathy 说：

> "人们把 prompt 和短任务描述联系在一起……但在每个**工业级** LLM 应用中，上下文工程才是真正的工作。"

换句话说：
- **Prompt Engineering** 是入门技能
- **Context Engineering** 是进阶技能

两者不是替代关系，而是包含关系。

## 三、为什么 2025-2026 年成为热门话题

### 3.1 模型窗口增长带来的新挑战

| 模型 | 上下文窗口 | 发布时间 |
|------|-----------|---------|
| GPT-4 | 128K | 2023 |
| Claude 3.5 Sonnet | 200K | 2024 |
| Gemini 1.5 Pro | 2M | 2024 |
| GPT-4.5 | 256K | 2025 |
| Claude 4 Opus | 200K | 2025 |

窗口变大了，但问题来了：**更多上下文 ≠ 更好的输出**。

Chroma 在 2025 年发布了一项研究：测试了 18 个模型，**全部**在输入增长时性能下降，部分模型准确率从 95% 降至 60%。

这个现象叫 **Context Rot（上下文腐烂）**——Transformer 的 n² 注意力机制导致 token 越多，召回越差。

### 3.2 AI Agent 的爆发

Agent 和传统 Chatbot 的区别：

```
Chatbot:  用户 → Prompt → LLM → 回答（单轮）
Agent:    用户 → 目标 → [规划 → 执行 → 观察 → 反思] × N轮 → 最终结果
```

Agent 需要在**长时间循环**中管理不断演变的信息状态，简单的 Prompt 不再足够。

### 3.3 "别再卷提示词"的共识

2025 年，业界达成了一个共识：**优化单条指令的收益已经递减**。

- Prompt Engineering 的红利期已过
- 系统级信息编排成为新的瓶颈
- 需要工程化的方法来管理上下文

### 3.4 Lost in the Middle 问题被证实

斯坦福和伯克利的研究发现：LLM 对输入**开头和结尾**注意力最强，**中间信息**注意力大幅下降，准确率可降低 30%+。

这意味着：即使你把所有信息都塞进上下文窗口，模型也可能"看不到"中间的内容。

### 3.5 2025 年被公认为"AI 第一技能"

OpenAI 产品负责人 Adithya Thatipalli、Anthropic 工程团队、LangChain 创始人等多方共识：上下文工程是 2025-2026 年 AI 应用开发的**第一技能**。

## 四、上下文工程的四大核心策略

LangChain 在 2025 年提出了一个被广泛采用的框架，把上下文工程归纳为四个策略：

```
┌─────────────────────────────────────────────────────────────┐
│                    Context Engineering                       │
├─────────────┬─────────────┬─────────────┬──────────────────┤
│    Write    │   Select    │  Compress   │     Isolate      │
│   （写入）   │   （选择）   │   （压缩）   │    （隔离）       │
├─────────────┼─────────────┼─────────────┼──────────────────┤
│ 存到上下文   │ 只拉取相关   │ 只保留必要   │ 拆分独立上下文    │
│ 窗口外部     │ 信息         │ token       │                  │
└─────────────┴─────────────┴─────────────┴──────────────────┘
```

### 4.1 Write（写入）—— 存到上下文窗口外部

**核心思想**：不是所有信息都需要实时在上下文里，可以先存起来，需要时再取。

**三种记忆类型**：

| 类型 | 类比 | 示例 |
|------|------|------|
| **情景记忆**（Episodic） | Few-shot 示例 | "上次用户问退货，我这样回答的" |
| **程序记忆**（Procedural） | 行为指引 | "遇到退款问题先查订单状态" |
| **语义记忆**（Semantic） | 事实与知识 | "用户的会员等级是 VIP" |

**实际案例**：
- **ChatGPT**：自动生成用户偏好记忆（"用户喜欢简洁的回答"）
- **Cursor**：学习你的编码模式，下次自动应用
- **Claude Code**：用 `CLAUDE.md` 文件存储项目级记忆

### 4.2 Select（选择）—— 只拉取当前步骤相关的信息

**核心思想**：上下文窗口有限，只放最相关的信息。

**主要手段**：
- **RAG 检索**：文档切块 → 向量化 → 检索 → 重排序
- **工具选择**：用 RAG 管理工具描述，选择准确率提升约 3 倍
- **混合搜索**：grep + embedding + re-ranking

**关键挑战**：精确度——不精确的检索会添加干扰项，反而降低输出质量。

**实际案例**：
- **Cursor/Windsurf**：结合 RAG 理解项目架构、跨文件依赖
- **Claude Code**：用 `head`、`tail` 分析大文件，而非全量加载

### 4.3 Compress（压缩）—— 只保留必要 token

**核心思想**：信息太多时，压缩比丢弃更好。

**压缩手段**：

| 手段 | 说明 | 示例 |
|------|------|------|
| **对话摘要** | 把长对话压缩成关键点 | Claude Code 在 95% 容量时触发 auto-compact |
| **工具输出压缩** | 缩减冗长的工具返回 | 把 10KB 的 API 响应压缩成 500 token |
| **修剪**（Trimming） | 删除旧消息 | 只保留最近 N 轮对话 |
| **训练摘要模型** | 专门训练的压缩模型 | Cognition（Devin）的做法 |

**实际案例**：
- **Claude Code**：auto-compact 机制，上下文快满时自动摘要
- **Devin**：训练专门的摘要模型管理上下文
- **Sentinel 框架**（2025.5）：轻量高效的上下文压缩方法

### 4.4 Isolate（隔离）—— 跨边界拆分上下文

**核心思想**：不同任务用不同的上下文窗口，避免信息污染。

**主要手段**：
- **多 Agent 架构**：子 Agent 各有独立干净的上下文窗口，只返回精炼摘要（1-2K tokens）
- **沙箱隔离**：代码 Agent 在沙箱执行，大状态存在环境变量中
- **State Schema 隔离**：只暴露必要字段给 LLM

**实际案例**：
- **Claude Code**：子 Agent 架构处理复杂研究任务
- **HuggingFace CodeAgent**：沙箱隔离模式，大状态存在执行环境中
- **Pokémon Agent**（Anthropic 研究）：通过结构化笔记跨数千步追踪游戏状态

## 五、上下文失败模式与对策

Drew Breunig 在 2025 年总结了四种上下文失败模式，这个框架被广泛引用：

### 5.1 Context Poisoning（上下文污染）

**问题**：幻觉或错误信息进入上下文，被反复引用，导致错误累积。

**场景**：
- Agent 在第一步产生了错误的推理
- 这个错误被写入记忆
- 后续所有步骤都基于这个错误继续

**对策**：
- 验证后再加入记忆
- 隔离信息类型（事实 vs 推测 vs 用户输入）
- 设置记忆的可信度标签

### 5.2 Context Distraction（上下文分散）

**问题**：模型过度关注累积的历史信息，忽略了自身的训练知识。

**场景**：
- 对话历史太长，模型沉迷于找历史答案
- 忽略了自己已经知道的通用知识

**对策**：
- 上下文摘要压缩
- 限制历史轮数
- 定期重置上下文

### 5.3 Context Confusion（上下文混淆）

**问题**：不相关信息导致输出变差。

**场景**：
- RAG 检索回来的文档和问题不太相关
- 工具列表太长，模型选错了工具

**对策**：
- 工具负载管理（RAG 选工具，≤30 个）
- 提高检索精度
- 过滤低相关度内容

### 5.4 Context Clash（上下文冲突）

**问题**：信息互相矛盾。

**研究数据**：Anthropic 的研究显示，上下文冲突可导致 **39% 性能下降**。

**场景**：
- 系统提示说"用中文回答"，但用户用英文提问
- 记忆中的用户偏好和当前指令矛盾

**对策**：
- 剪枝矛盾信息
- 使用 Anthropic think tool（提升 54%）
- 优先级机制（用户指令 > 系统提示 > 记忆）

## 六、主流框架与工具

### 6.1 框架对比

| 框架/工具 | 类型 | 核心能力 | 适用场景 |
|-----------|------|---------|---------|
| **LangGraph + LangSmith** | Agent 框架 | 状态管理、检查点、长期记忆、上下文摘要节点 | 复杂 Agent 工作流 |
| **LangMem** | 记忆库 | 高级记忆抽象（episodic/semantic/procedural） | 记忆密集型应用 |
| **Mem0** | 智能记忆 | 持久化、压缩、检索的记忆系统 | 通用记忆管理 |
| **LlamaIndex** | RAG 框架 | 高级记忆与工具选择技术 | 知识密集型应用 |
| **Sentinel** | 压缩框架 | 轻量级上下文压缩 | 长对话压缩 |
| **MCP** | 协议 | 模型上下文协议标准 | 工具与上下文标准化 |

### 6.2 选型建议

```
你的需求是什么？
│
├── 做 Agent 工作流 ──→ LangGraph
├── 做记忆管理 ──→ Mem0 或 LangMem
├── 做 RAG 系统 ──→ LlamaIndex
├── 做长对话压缩 ──→ Sentinel
└── 做工具标准化 ──→ MCP
```

## 七、实战案例

### 7.1 Claude Code 的上下文工程

Claude Code 是上下文工程的教科书级实现：

**Write**：
- `CLAUDE.md` 文件存储项目级记忆
- 用户偏好自动学习和持久化

**Select**：
- 用 `head`、`tail` 分析大文件，而非全量加载
- RAG 检索相关代码片段

**Compress**：
- auto-compact 机制：上下文达到 95% 时自动摘要
- 工具输出自动压缩

**Isolate**：
- 子 Agent 架构处理复杂研究任务
- 每个子 Agent 有独立的上下文窗口

### 7.2 Pokémon Agent（Anthropic 研究）

Anthropic 在 2025 年发布了一个用 Claude 玩 Pokémon 的研究，展示了上下文工程在长任务中的威力：

**挑战**：
- 游戏可能持续数千步
- 需要追踪地图、物品、任务进度
- 上下文窗口远远装不下所有信息

**解决方案**：
- 结构化笔记（NOTES.md）：记录关键发现
- 分层记忆：短期（当前房间）→ 中期（当前区域）→ 长期（全局地图）
- 动态加载：只在需要时加载相关区域的信息

### 7.3 Devin（Cognition）的摘要模型

Devin 是第一个 AI 软件工程师，它的上下文工程方案很独特：

**挑战**：
- 编码任务可能涉及数百个文件
- 调试过程可能产生大量日志
- 需要跨文件理解项目结构

**解决方案**：
- 训练专门的摘要模型（不是用通用 LLM 做摘要）
- 针对代码场景优化的压缩策略
- 渐进式加载：先看目录结构，再看具体文件

## 八、上下文工程的核心原则

### 原则 1：少即是多

不要把所有信息都塞进上下文窗口。研究证明，**精选的信息比海量的信息更有效**。

### 原则 2：动态组装

上下文不是静态的，应该根据当前任务动态组装。

```
用户问退货 → 加载退货政策 + 订单记录
用户问安装 → 加载安装指南 + 服务网点
用户闲聊   → 只保留对话历史
```

### 原则 3：分层管理

```
┌─────────────────────────────────────┐
│           系统提示（固定）            │  ← 每次都在
├─────────────────────────────────────┤
│         工具定义（半固定）            │  ← 根据任务选择
├─────────────────────────────────────┤
│       检索知识（动态）               │  ← RAG 按需加载
├─────────────────────────────────────┤
│       对话历史（滚动）               │  ← 保留最近 N 轮
├─────────────────────────────────────┤
│       工具输出（临时）               │  ← 用完即弃
└─────────────────────────────────────┘
```

### 原则 4：监控与迭代

上下文工程不是一劳永逸的，需要持续监控：
- 哪些信息被频繁使用？
- 哪些信息从未被用到？
- 上下文使用率是多少？
- 输出质量如何随上下文变化？

## 九、总结

**上下文工程的本质**：

> 在有限的上下文窗口中，为模型组装**最相关、最准确、最精简**的信息环境。

**与 Prompt Engineering 的关系**：

- Prompt Engineering 是上下文工程的**子集**
- Prompt 只是系统指令，上下文是**完整的信息环境**
- 两者是包含关系，不是替代关系

**四大核心策略**：

| 策略 | 一句话 | 代表工具 |
|------|--------|---------|
| **Write** | 存到外部，需要时取 | Mem0、LangMem |
| **Select** | 只拉取相关的 | RAG、向量数据库 |
| **Compress** | 压缩比丢弃好 | Sentinel、auto-compact |
| **Isolate** | 隔离不同任务 | 多 Agent、沙箱 |

**2025-2026 年的核心洞察**：

- 模型能力在提升，但上下文管理的难度也在提升
- 上下文工程是构建可靠 AI Agent 的**第一工程技能**
- 从"写好 Prompt"到"管理好上下文"，是每个 AI 开发者的进阶之路
